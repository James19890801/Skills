# 流程智能监控设计 Skill

## 核心交付物

1. **《流程智能监控方案》**（Markdown格式）：含指标定义、阈值设置、异常分类、根因分析框架、优化决策树、Owner报告模板的完整结构化文档
2. **可视化监控看板**（HTML格式）：带MOCK数据的交互式监控看板，员工一目了然，可直接在浏览器打开使用
   - 必须包含真实感的模拟数据（节点名称、人员姓名、具体数值）
   - 三色状态标识（红色报警/黄色预警/绿色正常）
   - 异常定位到具体节点和人员
   - AI建议直接展示在看板上
3. **预警报告示例**：模拟一份Owner会收到的分析报告样本，帮助理解方案落地效果

## 执行流程

```
用户描述流程名称与业务目标
    ↓
【步骤 1】信息采集
    - 流程名称与所属业务域
    - 流程核心目标（如：合同审批 → 目标：3天内完成审批）
    - 关键节点（2-5个主要审批/处理节点）
    - 当前已知的痛点（可选）
    ↓
【步骤 2】指标设计
    - 推荐2-4个核心监控KPI
    - 每个指标含：名称 / 定义 / 计算方式 / 正常范围 / 异常阈值
    - 优先推荐：时效类指标 + 质量类指标 + 异常类指标
    ↓
【步骤 3】异常规则设置
    - 超时类异常：超过时限X天/小时触发预警
    - 高驳回异常：驳回率超过阈值触发分析
    - 重复异常：同一实例被驳回≥N次触发深度分析
    - 断点异常：流程中断/挂起超过X小时触发报警
    - 用户可自定义补充规则
    ↓
【步骤 4】根因分析框架
    - 针对每类异常，给出标准排查思路
    - 例：高超时率 → 排查：节点处理人负荷 / 信息不完整 / 流程规则问题
    ↓
【步骤 5】优化决策树
    根据异常类型，给出决策路径：
    - 改规则（调整审批条件/阈值）
    - 改流程（调整节点顺序/增减节点）
    - 授权（增加审批人/设置代理规则）
    - 培训（执行者能力问题 → 培训）
    - 自动化（高置信度场景 → 自动审批）
    ↓
【步骤 6】输出监控方案
    生成完整《流程智能监控方案》文档（Markdown格式）
    ↓
【步骤 7】生成Owner预警报告模板
    基于方案，生成一份模拟的Owner预警通知示例
    ↓
【步骤 8】生成可视化监控看板（带MOCK数据）
    输出HTML格式的监控看板，预填充真实感的模拟数据，让员工一目了然：
    
    【必含模块】
    - 顶部KPI卡片：超时率/驳回率/转交率/处理量/异常数/响应率，带三色状态
    - 异常预警横幅：红色报警实时推送，含根因定位和AI建议
    - 流程健康度总览：多流程横向对比，超时/驳回/转交三列数据
    - 实时异常面板：按高/中/低优先级展示，每条含具体节点/人员/AI建议
    - 趋势图表：近7日数据柱状图，标注阈值线
    - 超时节点TOP排名：带节点名称、所属流程、具体时长
    - 改进任务追踪：待启动/进行中/已完成状态管理
    
    【MOCK数据要求】
    - 节点名称：使用真实业务术语（如"法务审核""财务审批""总经理审批"）
    - 人员姓名：使用常见中文姓名（如"张XX""李XX"）
    - 数值真实感：符合业务逻辑（如超时率32%、驳回率28%）
    - 异常描述：具体到原因（如"法务节点人手不足""合同条款不合规"）
    - AI建议：可落地的改进措施（如"授权增设二级法务"）
    ↓
【步骤 9】自动打开看板预览
    看板生成后，必须立即使用run_in_terminal工具启动本地服务并自动打开浏览器预览：
    - 命令：start "" "看板文件完整路径"
    - 无需用户确认或额外指令，确保交付即可见
```

## 标准监控指标体系

### 时效类指标
| 指标 | 定义 | 建议阈值参考 |
|------|------|------------|
| 平均审批时长 | 从提交到最终完成的平均天数 | 根据SLA定义 |
| 超时率 | 超过约定时限的实例占比 | >20%触发预警 |
| 节点滞留时长 | 单个节点平均等待时间 | >SLA的50%触发预警 |

### 质量类指标
| 指标 | 定义 | 建议阈值参考 |
|------|------|------------|
| 驳回率 | 被驳回实例占总数比例 | >30%触发异常 |
| 重复驳回率 | 同一实例被驳回≥2次的比例 | >10%触发深度分析 |
| 一次通过率 | 首次提交即通过的比例 | <60%需关注 |

### 异常类指标
| 指标 | 定义 | 建议阈值参考 |
|------|------|------------|
| 周驳回频次 | 同一流程一周内被驳回次数 | ≥3次触发异常标记 |
| 断点率 | 流程中断/挂起实例占比 | >5%触发排查 |
| 异常集中度 | 异常是否集中在特定节点/人员/时段 | 集中度>60%需定向分析 |

## 流程治理最佳实践（内置知识）

### 异常定义原则
1. **驳回异常**：同一流程实例一周内被驳回≥3次，定义为异常流程，需人工介入
2. **超时分级**：超时≤50%为黄色预警（催办），超时>100%为红色报警（升级处理）
3. **节点集中预警**：同一节点在同一时段异常数量>整体的60%，触发节点专项分析

### Owner收到的应该是报告，不是数字
- 错误方式：推送"超时率：38%"
- 正确方式：推送"合同审批超时率本周达38%，法务节点为主要卡点，建议授权二级法务处理标准合同"

### 优化决策优先级
1. 先排查规则问题（最容易改，成本最低）
2. 再看流程设计问题（需要重新建模）
3. 最后考虑执行者问题（需要培训）
4. 有条件时考虑自动化（高置信度场景）

## 反问用户的问题模板

当信息不足时，优先级顺序：

1. **流程基本信息**
   - "这个流程的核心业务目标是什么？（比如：3天内完成审批）"
   - "流程有哪几个主要节点？"

2. **当前痛点**
   - "目前这个流程最让人头疼的问题是什么？"
   - "有没有已知的卡点或高频问题？"

3. **Owner情况**
   - "谁是这个流程的负责人？他/她希望收到什么格式的报告？"

## 注意事项

1. **方案要可落地**：所有指标和阈值必须结合用户的实际业务场景，不能照搬模板数字
2. **Owner视角**：始终从流程Owner的角度设计报告，他/她收到的应该是"建议"而不是"数据"
3. **循序渐进**：建议先监控2-3个最关键指标，跑通之后再扩展，不要一次设计10个指标
4. **不混淆流程挖掘**：本Skill聚焦于设计监控规则，不做历史数据的回溯分析（那是process-mining的工作）
